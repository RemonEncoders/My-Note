# ১. প্রথমে Call Stack বুঝো

ধরো তোমার code:

```cs
public void Controller()
{
    Service();
}

public void Service()
{
    Repository();
}

public void Repository()
{
    throw new Exception("DB Failed");
}
```

Execution flow:

```cs
Controller()
   ↓
Service()
   ↓
Repository()
```

এখন CPU/runtime একটা stack maintain করে।

এটাকে বলে:

# Call Stack

---

# ২. Stack এ কী থাকে?

প্রতিবার method call হলে একটা stack frame create হয়।

ধরো:

```cs
| Repository Frame |
| Service Frame    |
| Controller Frame |
```

Top এ current executing method।

মানে এখন Repository চলছে।

---

# ৩. Normal Return কীভাবে হয়?

যদি exception না হয়:

```cs
Repository returns
	→ Service returns
		→ Controller returns
```

Stack cleanভাবে pop হয়।

```cs
Before Return:
| Repository |
| Service    |
| Controller |

After Repository Return:
| Service    |
| Controller |
```

এটা cheap।

Very optimized।

---

# ৪. Exception হলে কী হয়?

এখন:

```cs
throw new Exception();
```

হওয়ার সাথে সাথে CLR (.NET runtime) normal flow বন্ধ করে।

কারণ runtime এখন catch block খুঁজবে।

---

# ৫. কোথায় catch আছে?

ধরো:

```cs
public void Controller()
{
    try
    {
        Service();
    }
    catch(Exception ex)
    {
    
    }
}
```

কিন্তু Service/Repository এ catch নাই।

---

# ৬. এখন Runtime কী করে?

এখন CLR stack এর উপরের frame থেকে নিচে নিচে যেতে থাকে।

মানে: 

```cs
Repository
   ↑
Service
   ↑
Controller
```

এবং প্রতিটা frame remove করতে থাকে।

এই removing process-ই:

# Stack Unwinding

---

# ৭. Visualization

Before Exception:

```cs
	| Repository |
	| Service    |
	| Controller |
```

Exception thrown:

```cs
throw new Exception()
```

CLR বলে:

> "Repository এ catch নাই"

তাই Repository frame remove।

```cs
| Service    |
| Controller |
```

আবার check:

> "Service এ catch নাই"

Service frame remove।

```cs
| Controller |
```

এখন:

> "Controller এ catch আছে"

তাই এখানে execution jump করবে।

---

# ৮. কেন এটা expensive?

কারণ runtime কে:

- method frames inspect করতে হয়
- cleanup করতে হয়
- finally block run করতে হয়
- stack trace build করতে হয়
- metadata read করতে হয়

এটা normal return এর মতো simple না।

---

# ৯. Finally Block কেন important?

ধরো:

```cs
try
{
    Repository();
}
finally
{
    Console.WriteLine("Cleanup");
}
```

Exception হলেও finally run হবে।

মানে stack unwinding এর সময় CLR cleanup code execute করে।

এটাও cost add করে।

---

# ১০. Real Production Example

ধরো:

```cs
Controller
 → Service
   → Repository
      → SQL Driver
         → Network Layer
            → Socket Layer
```

Database fail করলো।

এখন exception bubble up করবে।

CLR multiple stack frame unwind করবে।

Large enterprise app এ hundreds of frames থাকতে পারে।

তাই exception expensive।

---

# ১১. Important Mental Model

Normal flow:

```cs
Method Return
```

Exception flow:

```cs
Emergency Escape System
```

Exception হচ্ছে emergency mechanism।

Daily business logic path না।

---

# ১২. Why Exception Should NOT Control Logic

BAD:

```cs 
try
{
    var item = list.First();
}
catch
{
    return null;
}
```

এখানে exception intentionally use হচ্ছে।

মানে runtime forced হচ্ছে stack unwind করতে।

Huge waste।

---

GOOD:

```
var item = list.FirstOrDefault();
```

No exception।  
No unwinding।

Fast।

---

# ১৩. Stack Trace কীভাবে তৈরি হয়?

যখন exception throw হয়:

```cs
throw new Exception();
```

CLR current stack snapshot নেয়।

Example:

```cs
	Repository()
	Service()
	Controller()
```

এটাই stack trace।

---

# ১৪. Why "throw ex" bad?

BAD:

```cs
catch(Exception ex){
	    throw ex;
}
```

এতে original stack trace reset হয়।

---

GOOD:

```cs
catch{
	    throw;
}
```

Original trace preserved।

---

# ১৫. Real Internal Analogy

Imagine building stairs।

```cs
	Floor 3 → Repository
	Floor 2 → Service
	Floor 1 → Controller
```

Normal return:

ধীরে ধীরে নিচে নামা।

Exception:

Fire alarm।

সব floor emergency exit দিয়ে দ্রুত বের হওয়া।

এই emergency evacuation = stack unwinding।

---

# ১৬. Why Middleware Good Here?

কারণ exception যত উপরে যাবে তত centralized হবে।

তুমি everywhere catch না দিয়ে:

```cs
	Repository
	Service
	Controller
	Middleware
```

শেষে middleware এ একবার catch করো।

Clean architecture।

---

# ১৭. Senior Engineer Mental Model

Exception মানে:

> "Something exceptional happened that broke normal execution flow."

এটা if-else alternative না।

---

# ১৮. Golden Rule

Use exceptions for:

✅ unexpected failures  
✅ infrastructure issues  
✅ unrecoverable problems

Avoid exceptions for:

❌ normal business flow  
❌ validation branching  
❌ existence checks  
❌ loop control

---

# ১৯. Ultra Important Insight

try-catch itself cheap।

BUT:

` THROW is expensive.`

কারণ throw ⇒ stack unwinding শুরু।

এটাই মূল কথা।
