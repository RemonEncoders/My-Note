# Scenario

ধরো:

```cs
public IActionResult Create()
{
    try
    {
        _orderService.CreateOrder();
        return Ok();
    }
    catch(Exception ex)
    {
        _logger.LogError(ex, "Controller Catch");
        return BadRequest();
    }
}
```

---

Service:

```cs
public void CreateOrder()
{
    try
    {
        Validate();
    }
    catch(Exception ex)
    {
        _logger.LogError(ex, "Service Catch");

        throw;
    }
}
```

---

Validation:

```cs
public void Validate()
{
    throw new Exception("Validation Failed");
}
```


> Exception যদি service layer এ throw হয়, তারপর service এ catch হয়, আবার controller এও catch থাকে — তাহলে stack trace, propagation, unwinding internally কীভাবে কাজ করে?”

---
# ১. Initial Call Stack

Execution শুরু:

```cs
Controller()   
	↓
CreateOrder()
   ↓
Validate()
```

Stack:

```cs
| Validate()      |
| CreateOrder()   |
| Controller()    |
```

---

# ২. Exception Throw হলো

```cs
throw new Exception("Validation Failed");
```

এখন CLR immediately normal flow stop করে।

---

# ৩. CLR প্রথমে current frame এ catch খুঁজবে

মানে:

```cs
Validate()
```

এখানে catch নাই।

তাই:

# Validate frame unwind হবে

Stack:

```cs
| CreateOrder() |
| Controller()  |
```

---

# ৪. এখন Service Layer এ পৌঁছালো

এখানে catch আছে:

```cs
catch(Exception ex){
    _logger.LogError(ex);
	throw;
}
```

তাই CLR এখানে থামবে।

Execution এখন catch block এ jump করবে।

---

# ৫. এখানে গুরুত্বপূর্ণ জিনিস

এই মুহূর্তে stack trace already generated।

মানে exception object internally remember করছে:

```cs
	Validate()
	CreateOrder()
	Controller()
```

---

# ৬. Service Catch এ কী হচ্ছে?

এখন:

```cs
_logger.LogError(ex);
```

এটা শুধু log করছে।

No problem।

---

# ৭. তারপর `throw;`

এখন সবচেয়ে important part।

```cs
throw;
```

মানে:

> "আমি exception handle করলাম না। উপরে পাঠাও।"

এখন CLR আবার unwinding continue করবে।

---

# ৮. Service Frame unwind হবে

কারণ throw; মানে rethrow।

Stack এখন:

```cs
| Controller() |
```

---

# ৯. Controller Catch এ পৌঁছাবে

এখন:

```cs
catch(Exception ex)
```

Execution এখানে আসবে।

---

# ১০. Final Stack Trace কী হবে?

IMPORTANT:

যেহেতু তুমি `throw;` ব্যবহার করেছো, original trace preserved থাকবে।

তাই stack trace:

```cs
	Exception: Validation Failed
	
	at Validate()
	at CreateOrder()
	at Controller()
```

সব থাকবে।

---

# ১১. যদি `throw ex;` করতে?

ধরো service এ:

```cs
catch(Exception ex){
    throw ex;
}
```

এখন disaster।

---

# ১২. কেন disaster?

কারণ CLR এটাকে NEW THROW হিসেবে treat করে।

মানে stack trace reset।

---

# ১৩. তখন stack trace হবে

```cs
at CreateOrder()at Controller()
```

OR sometimes only:

```cs
at CreateOrder()
```

মানে original source:

```cs
Validate()
```

হারিয়ে যাবে।

Production debugging nightmare।

---

# ১৪. Internal Runtime Flow

Visualize করি।

---

## Original Throw

```cs
Validate() 
 THROW
```

CLR:

```CS
	Unwind Validate frame
			↓
	Found catch in Service
			↓
	Execute catch block
			↓
		  throw;
			↓
	Continue unwinding
			↓
	Found catch in Controller
```

---

# ১৫. তাহলে কি multiple catch bad?

না।

But depends WHY তুমি catch করছো।

---

# ১৬. BAD Practice

```CS
catch(Exception ex){
   _logger.LogError(ex);
   throw;
}
```

everywhere।

---

কারণ:

- duplicate logging
- noisy logs
- same exception multiple times
- log pollution

---

# ১৭. Real Production Problem

ধরো:

```cs
Repository catch log
Service catch log
Controller catch log
Middleware catch log
```

এক exception → 4 logs।

Production monitoring hell।

---

# ১৮. Senior Engineer Rule

# Catch ONLY when you can do something meaningful.

Meaningful মানে:

✅ recover  
✅ retry  
✅ translate exception  
✅ rollback  
✅ fallback  
✅ enrich context

NOT:

❌ just logging everywhere

---

# ১৯. Proper Architecture

Modern architecture এ usually:

---

## Validation Layer

```cs
throw new ValidationException();
```

---

## Service Layer

No catch unless needed।

---

## Controller

Usually no catch।

---

## Global Middleware

Single centralized catch।

---

# ২০. Real Production Flow

```cs
	ValidationException thrown
	      ↓
	Bubble up
	      ↓
	Global Exception Middleware
	      ↓
	Convert to HTTP 400
	      ↓
	Structured log once
```

Clean।  
Centralized।  
Observable।

---

# ২১. Exception Translation (Advanced)

Sometimes service catches for abstraction।

Example:

```CS
catch(SqlException ex){
    throw new BusinessException("Database unavailable");
}
```

এটা GOOD।

কারণ low-level infra exception → business-friendly exception এ convert হচ্ছে।

---

# ২২. Important Mental Model

Exception propagation works like:

```CS
	THROW
	 ↓
	Search catch
	 ↓
	If not found: unwind frame
	 ↓
	Search next catch
	 ↓
	Repeat
```

---

# ২৩. Final Senior-Level Understanding

When exception happens:

### Runtime does 3 things:

1. Stop normal execution
2. Unwind stack frames
3. Search for matching catch block

---


# ২৬. Final Mental Picture

Imagine exception like fire alarm।

Each method = room।

When fire starts:

```CS
Room → Corridor → Floor → Exit Gate
```

Runtime exits room-by-room looking for someone who can handle the fire।

That searching + exiting process = stack unwinding + exception propagation।
