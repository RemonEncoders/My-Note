# ১. প্রথমে বুঝি: Exception আসলে কী?

ধরো request আসলো:

```cs
POST /api/orders
```

এরপর flow:

```cs
Controller
   ↓
Service
   ↓
Repository
   ↓
Database
```

যেকোনো layer এ error হতে পারে:

- DB connection fail
- null reference
- divide by zero
- validation issue
- timeout
- external API fail

এই unexpected problem-ই exception।

---

# ২. পুরানো Style (.NET Framework era thinking)

আগে আমরা everywhere এটা লিখতাম:

```cs
try{    

// business logic

}catch(Exception ex){

    _logger.LogError(ex.Message);
    
}
```

এটা small app এ okay।

কিন্তু enterprise system এ problem শুরু হয়।

---

# ৩. কেন everywhere try-catch খারাপ?

ধরো 500 API আছে।

প্রতিটায়:

```cs
try{


}
catch{

}
```

এখন problem:

---

## Problem 1 — Duplicate Code

সবখানে same logging।

```cs
_logger.LogError(ex.Message);
```

DRY principle break।

---

## Problem 2 — Inconsistent Response

এক API return করছে:

```cs
{  "message": "Error"}
```

আরেকটা:

```cs
{  "error": "Something went wrong"}
```

আরেকটা plain text।

Frontend nightmare।

---

## Problem 3 — Developer ভুলে যাবে

নতুন developer try-catch দিল না।

Production crash।

---

## Problem 4 — Business Logic polluted

Controller clean থাকার কথা:

```cs
CreateOrder()
```

কিন্তু হয়ে গেল:

```cs
try{   

	...
	
}catch{

   ...
}
```

মানে business logic + infrastructure concern mixed।

---

## Problem 5 — Stack Trace Lost

অনেকে করে:

```cs
catch(Exception ex){
    throw ex;
}
```

এতে original stack trace নষ্ট হয়।

Production debugging hell।

---

# ৪. Modern Thinking (.NET Core Architecture)

.NET Core completely changed mindset।

এখানে concept:

# Cross Cutting Concern

যে জিনিস সব request এর জন্য common:

- Logging
- Authentication
- Authorization
- Exception Handling
- Rate Limiting
- Caching
- Metrics

এসব business logic এর মধ্যে না রেখে centralized pipeline এ handle করা হয়।
